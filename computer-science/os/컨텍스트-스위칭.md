# 컨텍스트 스위칭 (Context Switching)

## 핵심 개념

**Context Switching**이란 CPU 코어가 현재 실행 중인 프로세스/스레드를 멈추고 다른 프로세스/스레드로 전환할 때, 실행 상태를 저장하고 복원하는 과정이다. 멀티태스킹 환경에서 여러 작업을 번갈아 실행하기 위해 반드시 필요하지만, 그 자체로 오버헤드가 발생한다.

## 동작 원리

### Context란?

CPU가 현재 실행 상태를 나타내는 정보의 집합:

| 구성 요소 | 설명 |
|-----------|------|
| **PC (Program Counter)** | 다음에 실행할 명령어의 주소 |
| **CPU Registers** | 범용 레지스터들 (연산 중간값, 주소값 등) |
| **SP (Stack Pointer)** | 현재 스택의 위치 |

이 정보들을 통칭하여 **CPU context**라고 한다.

### 전환 과정

```
[Thread-A 실행 중]
  Time Quantum 만료 or I/O 요청 발생

  1. Thread-A의 CPU context (PC, SP, Registers)를 PCB/TCB에 저장
  2. OS 스케줄러가 다음 실행할 Thread-B를 선택
  3. Thread-B의 PCB/TCB에서 CPU context를 복원
  4. Thread-B 실행 재개
```

- **PCB (Process Control Block)** — OS가 프로세스마다 생성하는 자료구조. 프로세스의 context를 보관.
- **TCB (Thread Control Block)** — OS가 스레드마다 생성하는 자료구조. 스레드의 context를 보관.

### Context Switching이 발생하는 시점

- **Time Quantum 만료** — 멀티태스킹 OS가 시간 할당량을 다 쓴 스레드를 강제 전환
- **I/O 요청** — 스레드가 I/O 대기에 들어가면 CPU를 다른 스레드에게 양보
- **인터럽트** — 하드웨어/소프트웨어 인터럽트 발생 시
- **우선순위** — 더 높은 우선순위의 스레드가 ready 상태가 될 때

**스레드 컨텍스트 스위칭이 더 빠른 이유:**

같은 프로세스 내 스레드들은 **Code, Data, Heap** 영역을 공유한다. 따라서 CPU registers, PC, SP만 교체하면 된다.

반면 프로세스 간 전환은 메모리 주소 공간 자체가 다르기 때문에 추가 비용이 발생한다:

**1. MMU 매핑 변경**
- 가상 주소 → 물리 주소 매핑 정보를 새 프로세스 것으로 교체

**2. TLB (Translation Lookaside Buffer) Flush**
```
프로세스 A 실행 중:
  TLB: [A의 가상주소 → 물리주소 매핑 캐시]  ← 따뜻한 캐시

프로세스 B로 전환:
  TLB flush (전체 무효화)  ← A의 매핑은 B에서 무의미

프로세스 B 실행:
  TLB miss 연속 발생 → 페이지 테이블 재조회 필요
```

TLB는 가상 주소 → 물리 주소 변환을 캐싱하는 CPU 내부 캐시다. 프로세스가 바뀌면 매핑 자체가 달라지므로 통째로 비워야 하고, 이후 TLB miss가 연속 발생하면서 성능이 떨어진다.

**3. CPU 캐시 Cold Miss**
- L1/L2 캐시에 쌓인 이전 프로세스의 데이터가 무용지물
- 새 프로세스의 데이터를 다시 올리는 과정에서 cache miss 발생
