# 다이나믹 프록시 (Dynamic Proxy)

## 핵심 개념

**다이나믹 프록시**는 프록시 객체를 컴파일 타임이 아닌 **런타임에 동적으로 생성**하는 기술이다. 일반 프록시는 대상 클래스마다 프록시 클래스를 직접 작성해야 하지만, 다이나믹 프록시는 **하나의 핸들러로 여러 타입의 프록시를 생성**할 수 있다. Spring AOP, @Transactional, JPA 지연 로딩 등 프레임워크의 핵심 기반 기술이다.

## 동작 원리

### 일반 프록시 vs 다이나믹 프록시

```
일반 프록시 (Static Proxy):
  대상 클래스마다 프록시 클래스를 직접 작성
  UserServiceProxy.java     ← 직접 작성
  OrderServiceProxy.java    ← 직접 작성
  PaymentServiceProxy.java  ← 직접 작성
  → 대상이 늘어날수록 프록시 클래스도 증가

다이나믹 프록시 (Dynamic Proxy):
  런타임에 Proxy 객체 자동 생성
  InvocationHandler 하나로 모든 프록시 처리
  → 대상이 늘어나도 핸들러는 하나
```

### JDK Dynamic Proxy — 인터페이스 기반

`java.lang.reflect.Proxy`를 사용하며, **인터페이스가 반드시 필요**하다.

```
호출 흐름:
Client → Proxy (자동 생성) → InvocationHandler.invoke() → Target
                                    │
                              리플렉션으로 메서드 실행
                              전/후처리 추가 가능
```

```java
// 1. 인터페이스 정의
interface UserService {
    User findById(Long id);
}

// 2. 실제 구현
class UserServiceImpl implements UserService {
    public User findById(Long id) { return userRepo.findById(id); }
}

// 3. InvocationHandler — 프록시 동작 정의
class LogHandler implements InvocationHandler {
    private final Object target;

    LogHandler(Object target) { this.target = target; }

    @Override
    public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
        System.out.println("[LOG] " + method.getName() + " 호출");  // 전처리
        Object result = method.invoke(target, args);                // 원본 실행
        System.out.println("[LOG] " + method.getName() + " 완료");  // 후처리
        return result;
    }
}

// 4. 프록시 생성
UserService proxy = (UserService) Proxy.newProxyInstance(
    UserService.class.getClassLoader(),
    new Class[]{UserService.class},  // 인터페이스
    new LogHandler(new UserServiceImpl())
);

proxy.findById(1L);  // LogHandler.invoke() 경유
```

### CGLIB — 클래스 기반 (바이트코드 조작)

인터페이스가 **없어도** 프록시를 생성할 수 있다. 대상 클래스를 **상속**한 서브클래스를 바이트코드 레벨에서 동적으로 만든다.

```
JDK Dynamic Proxy:                CGLIB:
인터페이스 필수                     인터페이스 불필요
Proxy → InvocationHandler         Proxy(서브클래스) → MethodInterceptor
리플렉션으로 실행                   바이트코드 조작으로 실행 (더 빠름)
```

### Spring AOP에서의 선택

Spring은 상황에 따라 **자동으로** 프록시 방식을 선택한다.

```
@Transactional, @Cacheable, @Async 등 AOP 적용 시:

  대상 Bean에 인터페이스가 있다?
    YES → JDK Dynamic Proxy 사용
    NO  → CGLIB 사용

  Spring Boot 2.0+ 기본값:
    → CGLIB 우선 사용 (proxyTargetClass=true)
```

```
Controller → Proxy(AOP) → @Transactional 메서드
                │
                ├─ 트랜잭션 시작 (전처리)
                ├─ 실제 메서드 실행
                └─ 트랜잭션 커밋/롤백 (후처리)
```

### 프록시 방식의 한계 — self-invocation

프록시는 **외부에서** 호출할 때만 동작한다. 같은 객체 내부에서 `this.method()` 호출은 프록시를 거치지 않는다.

```java
@Service
class OrderService {
    @Transactional
    public void createOrder() { ... }

    public void process() {
        this.createOrder();  // self-invocation → 프록시 우회!
                             // @Transactional 동작 안 함
    }
}
```

```
외부 호출:    Client → Proxy → createOrder()  ← AOP 동작 ✓

self-invocation: process() → this.createOrder()
                                 ↑
                            프록시가 아닌 실제 객체의 this
                            → AOP 동작 안 함 ✗
```

## 코드 예시

```java
// CGLIB 사용 예시
Enhancer enhancer = new Enhancer();
enhancer.setSuperclass(UserServiceImpl.class);
enhancer.setCallback((MethodInterceptor) (obj, method, args, proxy) -> {
    System.out.println("[LOG] " + method.getName() + " 호출");
    Object result = proxy.invokeSuper(obj, args);  // 부모(원본) 메서드 호출
    System.out.println("[LOG] " + method.getName() + " 완료");
    return result;
});

UserServiceImpl proxy = (UserServiceImpl) enhancer.create();
proxy.findById(1L);
```
