# 리플렉션 (Reflection)

## 핵심 개념

**리플렉션(Reflection)**은 컴파일 타임에 알 수 없는 클래스의 정보를 **런타임에 동적으로 조회하고 조작**하는 기능이다. 클래스 이름, 필드, 메서드, 어노테이션 등의 메타데이터에 접근할 수 있으며, private 멤버까지 강제 접근이 가능하다. Spring DI, JPA, JUnit 등 대부분의 프레임워크가 리플렉션에 의존한다.

## 동작 원리

### 리플렉션이 가능한 이유

JVM은 클래스를 로딩할 때 해당 클래스의 메타정보를 **`java.lang.Class` 객체**로 만들어 Method Area에 저장한다. 리플렉션은 이 Class 객체를 통해 런타임에 클래스 구조를 탐색한다.

```
소스 코드 → 컴파일 → .class 파일 → ClassLoader가 JVM에 적재
                                         │
                                    Class<?> 객체 생성
                                    (Method Area에 저장)
                                         │
                                    리플렉션 API로 접근
                                    - 필드 목록
                                    - 메서드 목록
                                    - 생성자 목록
                                    - 어노테이션 목록
```

### Class 객체 얻기 (3가지 방법)

```java
// 1. Class.forName() — 문자열로 동적 로딩
Class<?> clazz = Class.forName("com.example.MyService");

// 2. .class 리터럴 — 컴파일 타임에 알고 있을 때
Class<MyService> clazz = MyService.class;

// 3. getClass() — 인스턴스에서
MyService instance = new MyService();
Class<?> clazz = instance.getClass();
```

### 리플렉션으로 할 수 있는 것

```
                    Class<?> 객체
                         │
         ┌───────────────┼───────────────┐
         │               │               │
    ┌────┴────┐    ┌─────┴─────┐   ┌─────┴──────┐
    │  Field  │    │  Method   │   │Constructor │
    │ 필드 접근 │    │ 메서드 실행 │   │ 인스턴스 생성│
    └─────────┘    └───────────┘   └────────────┘
    getDeclaredFields()  getDeclaredMethods()  getDeclaredConstructors()
    get() / set()        invoke()              newInstance()
    setAccessible(true)  setAccessible(true)
```

### Spring DI에서의 활용

Spring이 `@Autowired`를 처리하는 과정:

```
1. 컴포넌트 스캔 → @Component가 붙은 클래스 발견
2. Class.forName()으로 클래스 로딩
3. getDeclaredConstructor().newInstance()로 인스턴스 생성
4. getDeclaredFields()로 필드 목록 조회
5. @Autowired 어노테이션이 붙은 필드 발견
6. field.setAccessible(true)로 private 접근 허용
7. field.set(instance, bean)으로 Bean 주입
```

이처럼 리플렉션 덕분에 개발자가 직접 `new`로 생성하지 않아도 프레임워크가 알아서 객체를 만들고 의존성을 주입할 수 있다.

### 리플렉션의 단점

| 단점 | 원인 |
|------|------|
| **성능 오버헤드** | 런타임 메타데이터 탐색 비용, JIT 최적화 불가 |
| **타입 안전성 없음** | 컴파일 타임 체크 불가 → 런타임 오류 발생 가능 |
| **캡슐화 위반** | `setAccessible(true)`로 private 강제 접근 |

> [!warning] 성능 오버헤드의 원인
> 일반 메서드 호출은 컴파일 시 이미 결정되어 바로 실행되지만, 리플렉션은 런타임에 문자열로 메서드를 찾고, 접근 권한을 확인하고, 인자 타입을 검증하는 과정이 추가된다. JIT 컴파일러도 리플렉션 호출은 인라이닝 등의 최적화를 적용하지 못한다.

## 코드 예시

```java
// 1. 클래스 정보 조회
Class<?> clazz = Class.forName("com.example.User");
System.out.println(clazz.getName());           // com.example.User
System.out.println(clazz.getSimpleName());     // User

// 2. 인스턴스 생성
Object user = clazz.getDeclaredConstructor().newInstance();

// 3. private 필드 접근
Field nameField = clazz.getDeclaredField("name");
nameField.setAccessible(true);          // private 접근 허용
nameField.set(user, "홍길동");           // 값 설정
String name = (String) nameField.get(user);  // 값 읽기

// 4. 메서드 실행
Method greet = clazz.getDeclaredMethod("greet", String.class);
greet.setAccessible(true);
greet.invoke(user, "반갑습니다");  // user.greet("반갑습니다") 실행

// 5. 어노테이션 조회
if (clazz.isAnnotationPresent(Component.class)) {
    Component anno = clazz.getAnnotation(Component.class);
    System.out.println(anno.value());
}
```
