# 모니터 (Monitor)

## 핵심 개념

**Monitor**란 Race Condition을 방지하기 위해 Java가 제공하는 동기화 메커니즘으로, Lock과 Wait Set을 결합하여 Critical Section에 대한 상호 배제와 조건 대기를 함께 처리한다.

Java의 모든 객체는 **Monitor**를 내장하고 있다. `synchronized` 키워드가 이 Monitor를 활성화한다.

## 모니터 구성 요소

Monitor는 세 가지 핵심 구성 요소로 이루어진다:

| 구성 요소 | 역할 |
|---|---|
| **Mutex Lock** | Critical Section 진입 권한. 한 번에 하나의 스레드만 보유 가능 |
| **Entry Set** | 락 획득을 대기 중인 스레드들의 집합 |
| **Wait Set** | `wait()` 호출 후 조건을 대기 중인 스레드들의 집합 |

### notify() vs notifyAll()

| | `notify()` | `notifyAll()` |
|---|---|---|
| 대상 | Wait Set에서 **임의 1개** | Wait Set **전부** |
| 선택 가능 | 어떤 스레드가 깨어날지 **보장 불가** | 모두 깨움 |
| 위험 | **Starvation** — 특정 스레드가 계속 선택 안 될 수 있음 | 불필요한 경쟁 (성능 비용 낮음) |
| 권장 | 모든 스레드가 동일 조건일 때만 | **일반적으로 권장** |

### wait()에서 if 대신 while을 써야 하는 이유

```java
// ❌ 위험한 코드
synchronized (lock) {
    if (조건 불만족) {  // 한 번만 체크
        wait();
    }
    // 조건이 다시 깨졌을 수 있는데 그냥 진행 → 버그
}

// ✅ 안전한 코드
synchronized (lock) {
    while (조건 불만족) {  // 깨어날 때마다 재확인
        wait();
    }
    // 조건 확실히 만족된 상태에서 진행
}
```

`notifyAll()`은 모든 스레드를 깨우므로, 깨어난 스레드가 조건에 맞지 않을 수 있다. `while`로 재확인해야 **Spurious Wakeup**에도 안전하다.

### Biased Lock

대부분의 락은 실제로 **한 스레드만 반복 획득**하는 패턴이다. Biased Lock은 이를 최적화한다.

```
첫 번째 획득:
  Mark Word에 Thread ID 기록 (CAS 1회)

이후 재진입:
  Mark Word의 Thread ID == 현재 스레드? → 그냥 진입 (비용 0)
```

> [!note] Biased Lock과 Java 15+
> JDK 15부터 Biased Locking이 **deprecated** 되었고, JDK 18에서 기본 비활성화되었다. 현대 JVM에서는 Lightweight Lock부터 시작한다. 이유: Biased Lock의 revocation 비용이 실제 워크로드에서 이점을 상쇄하는 경우가 많았기 때문.

### Lightweight Lock — CAS (Compare-And-Swap)

다른 스레드가 진입을 시도하면 Biased Lock이 해제되고, **CAS 기반 스핀**으로 전환된다.

**CAS 동작 원리:**

```
CAS(메모리주소, 기대값, 새값)
  현재값 == 기대값 → 새값으로 교체 (성공, 원자적)
  현재값 != 기대값 → 실패 → 재시도
```

CAS는 **하드웨어 레벨 원자적 연산**으로, 락 없이 Thread-safe를 달성한다. Java에서는 `AtomicInteger`, `AtomicLong`, `AtomicReference` 등이 내부적으로 CAS를 사용한다.

```java
AtomicInteger count = new AtomicInteger(0);
count.incrementAndGet();  // 내부: CAS(현재값, 현재값+1) 반복
```

### Heavyweight Lock

CAS 스핀이 한계를 초과하면 OS 뮤텍스로 전환된다. 이 단계에서 Monitor의 Entry Set, Wait Set이 실제로 활성화되며 **컨텍스트 스위칭**이 발생한다.

## 코드 예시

### synchronized 메서드

```java
public class Counter {
    private int count = 0;

    // 락 = this (Counter 인스턴스)
    public synchronized void increment() {
        count++;
    }
}
```

### wait() / notifyAll() — Producer-Consumer

```java
public class BoundedBuffer {
    private final Queue<Integer> buffer = new LinkedList<>();
    private final int capacity;

    public BoundedBuffer(int capacity) {
        this.capacity = capacity;
    }

    public synchronized void produce(int item) throws InterruptedException {
        while (buffer.size() == capacity) {  // if 아닌 while!
            wait();  // 락 반납 + Wait Set 진입
        }
        buffer.add(item);
        notifyAll();  // Wait Set 전체 → Entry Set
    }

    public synchronized int consume() throws InterruptedException {
        while (buffer.isEmpty()) {
            wait();
        }
        int item = buffer.poll();
        notifyAll();
        return item;
    }
}
```
