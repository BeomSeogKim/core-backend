# 코루틴 기초: suspend, Continuation, 상태머신

## 핵심 개념

**Coroutine**은 스레드를 blocking하지 않고 일시 중단(suspend)할 수 있는 경량 실행 단위다. 컴파일러가 `suspend` 함수를 **Continuation** 기반 **상태머신(State Machine)**으로 변환하여, 스레드 없이도 실행 상태를 힙에 보존하고 재개할 수 있다.

> [!note] suspend vs blocking
> **blocking**은 스레드 자체가 대기하여 다른 작업을 수행할 수 없는 반면, **suspend**는 스레드를 해방시키고 코루틴만 일시 중단한다. 스레드는 다른 코루틴을 실행할 수 있다.

## 동작 원리

### suspend vs blocking

#### blocking
스레드 자체가 응답을 받을 때까지 아무것도 못하고 대기

```kotlin
// Thread.sleep → 스레드 blocking
Thread.sleep(1000)  // 이 스레드는 1초간 아무것도 못함
```

#### suspend
스레드는 해방시키고, 코루틴만 일시 중단. 스레드는 다른 코루틴 실행 가능.

```kotlin
// delay → 코루틴 suspend
delay(1000)  // 스레드는 해방됨, 다른 코루틴이 이 스레드 사용 가능
```

#### awaitSingle()이 문제가 되는 패턴

```kotlin
// 루프 내 awaitSingle() → 직렬 실행
chunks.forEach { chunk ->
    val result = productClient.getProducts(chunk).awaitSingle()
    // suspend이지만, 이 코루틴은 응답 후에만 다음 chunk로 진행
    // 결과: chunk 수 × 응답시간 만큼 누적
}
// 120 chunks × 250ms = 30초
```

스레드는 blocking되지 않지만, **이 코루틴**의 진행이 직렬로 강제됨.

> [!warning] 루프 내 awaitSingle() 주의
> `awaitSingle()`은 스레드를 blocking하지 않지만, 코루틴의 진행 자체는 직렬로 강제된다. 병렬 처리가 필요하면 [[코루틴-동시성]]의 Scatter-Gather 패턴을 사용해야 한다.

### Continuation: 코루틴의 핵심

#### suspend 함수 컴파일 결과

컴파일러가 `suspend` 함수를 **상태머신(State Machine)**으로 변환한다.

```kotlin
// 원본 코드
suspend fun fetchProduct(id: Long): Product {
    val detail = productClient.getDetail(id).awaitSingle()  // [상태 1]
    val price = priceClient.getPrice(id).awaitSingle()       // [상태 2]
    return Product(detail, price)
}
```

```kotlin
// 컴파일러가 생성하는 의사코드
fun fetchProduct(id: Long, continuation: Continuation<Product>) {
    val cont = continuation as? FetchProductContinuation
        ?: FetchProductContinuation(continuation)

    when (cont.state) {
        0 -> {
            cont.state = 1
            // 비동기 요청 시작 후 즉시 반환 (스레드 해방)
            productClient.getDetail(id).subscribe(cont)
            return COROUTINE_SUSPENDED
        }
        1 -> {
            val detail = cont.result as Detail
            cont.detail = detail     // 힙에 중간 결과 저장
            cont.state = 2
            priceClient.getPrice(id).subscribe(cont)
            return COROUTINE_SUSPENDED
        }
        2 -> {
            val price = cont.result as Price
            cont.resume(Product(cont.detail, price))
        }
    }
}
```

#### Continuation 객체가 저장하는 것

```
Continuation {
    state: Int          // 현재 어느 suspend point까지 실행했는지
    result: Any?        // 마지막 suspend 결과값
    detail: Detail?     // 중간 지역변수 (힙에 저장!)
    context: CoroutineContext
    completion: Continuation  // 부모 코루틴
}
```

**핵심**: 원래 스레드 스택에 있어야 할 지역변수들이 **힙의 Continuation 객체**에 저장된다.
- 스레드가 없어도 실행 상태를 보존할 수 있음
- 재개 시 어느 스레드에서든 실행 가능

#### 실행 흐름

```
[요청 진입]
  코루틴 시작 → state=0

[state=0 실행]
  productClient.getDetail() 호출
  → Continuation 힙에 저장 (state=1, 중간변수)
  → COROUTINE_SUSPENDED 반환
  → 스레드 해방 → Dispatcher에게 반납

[네트워크 응답 도착]
  → Dispatcher가 가용 스레드 배정
  → cont.resume(detail) 호출
  → state=1 실행 재개 (반드시 같은 스레드일 필요 없음)

[state=1 실행]
  detail을 cont.detail에 저장
  priceClient.getPrice() 호출
  → 또 suspend
  → 스레드 해방

[두 번째 응답 도착]
  → state=2 실행
  → Product 객체 생성하여 상위 코루틴에 resume
```

### OS Thread와의 근본적 차이

#### OS Thread 일시 중단

```
스레드가 I/O 대기 시:
  User Space → Kernel Space (syscall)
    현재 스레드 CPU 레지스터 저장 → TCB에 기록
    스케줄러가 다음 스레드 선택
    다음 스레드 레지스터 복원
  Kernel Space → User Space

  스레드는 OS 스케줄러가 깨울 때까지 대기
  CPU 점유는 없지만 메모리(~1MB 스택)는 계속 점유
```

#### Coroutine suspend

```
코루틴이 suspend 시:
  Continuation 객체에 상태 저장 (힙, ~수백 bytes)
  스레드 Dispatcher에게 반납 → 즉시 다른 코루틴 실행

  커널 진입 없음, OS 스케줄러 개입 없음
  스레드 메모리 점유 없음
```

> [!tip] Coroutine vs OS Thread 비용 차이
> Coroutine suspend는 커널 진입 없이 힙에 상태를 저장하므로 ~수십 ns 수준이다. 반면 OS Thread의 Context Switching은 커널 전환 + 캐시 오염으로 ~1~10μs가 소요된다.

| | OS Thread 대기 | Coroutine suspend |
|---|---|---|
| 커널 진입 | 필요 | 불필요 |
| 상태 저장 위치 | TCB (OS 관리) | Continuation 힙 객체 |
| 스레드 점유 | 계속 점유 | 즉시 해방 |
| 전환 비용 | ~1~10μs + 캐시 오염 | ~수십 ns |
| 메모리 | ~1MB 스택 | ~수백 bytes |
