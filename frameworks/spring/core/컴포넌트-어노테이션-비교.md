# 컴포넌트 어노테이션 비교

## 핵심 개념

`@Component`는 Spring이 컴포넌트 스캔을 통해 Bean으로 등록하는 기본 어노테이션이다. `@Service` / `@Repository` / `@Controller`는 모두 `@Component`를 **메타 어노테이션**으로 포함하고 있어 동일하게 Bean 등록 대상이 되며, 각 계층의 역할과 부가 기능을 추가로 제공한다.

## 동작 원리

### 메타 어노테이션

Java 어노테이션은 상속(`extends`)이 없다. 대신 어노테이션 위에 어노테이션을 붙이는 **메타 어노테이션** 방식으로 역할을 위임한다.

```java
@Target(ElementType.TYPE)
@Retention(RetentionPolicy.RUNTIME)
@Documented
@Component          // ← @Component를 메타 어노테이션으로 포함
public @interface Service {
    @AliasFor(annotation = Component.class)
    String value() default "";
}
```

Spring의 컴포넌트 스캔은 `@Component`가 **직접 붙었거나 메타 어노테이션으로 포함된** 경우를 모두 스캔 대상으로 인식한다.

### 어노테이션별 역할

```
        @Component (기반)
       /     |      \      \
 @Service @Repository @Controller @Configuration
```

| 어노테이션 | 계층 | 부가 기능 |
|---|---|---|
| `@Component` | 범용 | 없음 |
| `@Service` | 비즈니스 | 없음 (의미론적 마킹) |
| `@Repository` | 영속성 | **PersistenceException 변환** |
| `@Controller` | 표현 | Spring MVC HandlerMapping 인식 |
| `@Configuration` | 설정 | CGLIB 프록시로 @Bean 싱글톤 보장 |

### @Repository의 Exception Translation

`PersistenceExceptionTranslationPostProcessor`(BeanPostProcessor)가 `@Repository`가 붙은 Bean을 감지하여 프록시로 감싼다. 이 프록시가 `SQLException` 등 DB 벤더별 예외를 Spring의 `DataAccessException` 계층으로 변환한다.

```
Bean 생성
    ↓
PersistenceExceptionTranslationPostProcessor 감지
    ↓
@Repository Bean → Proxy로 교체
    ↓
SQLException → DataAccessException (Spring 추상 예외)
```

이를 통해 JPA, JDBC, MongoDB 등 영속성 기술이 바뀌어도 예외 처리 코드를 변경하지 않아도 된다.

## @Service를 쓰는 이유

부가 기능이 없음에도 `@Service`를 별도로 쓰는 이유:

1. **계층 명확성** — 코드만 봐도 비즈니스 레이어임을 즉시 파악
2. **AOP 포인트컷 타겟팅** — `@Service` 기준으로 AOP를 적용하면 서비스 계층에만 선택적으로 Advice를 적용 가능

```java
// 서비스 계층 전체에 실행 시간 로깅 AOP 적용
@Around("@within(org.springframework.stereotype.Service)")
public Object logExecutionTime(ProceedingJoinPoint joinPoint) throws Throwable {
    // ...
}
```

3. **향후 확장성** — Spring이 나중에 `@Service`에 기능을 추가할 여지
