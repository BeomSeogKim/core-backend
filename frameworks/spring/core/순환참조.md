# 순환참조 (Circular Dependency)

## 핵심 개념

**순환참조**는 Bean A가 Bean B에 의존하고, Bean B가 다시 Bean A에 의존하는 구조다. 이는 설계가 잘못됐다는 신호로, 두 클래스가 너무 강하게 결합되어 있음을 의미한다.

```
A → B → C → A  (순환)

또는 단순:
A ⇄ B  (양방향 의존)
```

## 동작 원리

### 생성자 주입 vs 필드/세터 주입의 차이

| | 생성자 주입 | 필드/세터 주입 |
|---|---|---|
| 감지 시점 | **앱 시작 시 즉시** 실패 | Spring Boot 2.6+: 시작 시 실패 / 이전: 런타임 |
| 에러 | `BeanCurrentlyInCreationException` | `BeanCurrentlyInCreationException` (또는 숨겨짐) |
| 장점 | Fast-fail — 문제를 조기에 발견 | — |

생성자 주입은 Bean을 생성할 때 즉시 의존성이 필요하므로, A를 만들려면 B가 필요하고 B를 만들려면 A가 필요한 순간 Spring이 즉시 실패한다.

```
Spring Boot 2.6+: spring.main.allow-circular-references=false (기본)
→ 주입 방식 무관하게 시작 시 순환참조 금지
```

### 해결방법

```
우선순위: 1순위 설계 수정 → 2순위 임시방편
```

**1순위: 의존성 역전 (DIP)**

인터페이스를 도입하거나 중간 서비스를 추출해 순환을 끊는다.

```
Before: A ⇄ B

After:  A → C ← B   (공통 의존성 C로 분리)
```

```java
// Before: OrderService ⇄ PaymentService
// After: 공통 로직을 EventPublisher로 분리
@Service
public class OrderService {
    private final ApplicationEventPublisher eventPublisher;
    // PaymentService 직접 의존 제거
}
```

**2순위: @Lazy (비권장)**

한쪽을 지연 초기화해 프록시로 감싼다. 순환 구조는 그대로 유지되므로 임시방편이다.

```java
@Service
public class A {
    public A(@Lazy B b) { ... }  // B를 프록시로 감싸 실제 사용 시점에 초기화
}
```

**3순위: 세터 주입으로 전환 (비권장)**

객체 생성 후 의존성을 주입하므로 순환참조를 회피할 수 있다. 역시 설계 문제는 그대로다.

```java
@Service
public class A {
    private B b;

    @Autowired
    public void setB(B b) { this.b = b; }
}
```

### @Lazy / 세터 주입이 비권장인 이유

순환참조가 발생했다는 것 자체가 두 클래스가 양방향으로 결합되어 있다는 뜻이다. 에러만 숨길 뿐 구조적 결함은 그대로여서 런타임 무한 루프, 예상치 못한 초기화 순서 등의 문제가 잠재한다.
