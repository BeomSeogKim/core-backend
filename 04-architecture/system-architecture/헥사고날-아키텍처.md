# 헥사고날 아키텍처 (Hexagonal Architecture)

## 핵심 개념

애플리케이션의 **핵심 비즈니스 로직(Domain)** 을 외부 세계로부터 완전히 격리하는 아키텍처이다. **Ports and Adapters** 패턴이라고도 불린다.

```
[외부 Actor]  →  Adapter  →  Port  →  Application  →  Port  →  Adapter  →  [외부 시스템]
   (요청)        (변환)     (인터페이스)  (비즈니스 로직)  (인터페이스)  (구현체)
```

> [!note] 핵심 원칙
> 외부 의존성(DB, HTTP, 메시지 큐)은 반드시 Adapter를 통해서만 접근하고, Application Layer는 Port(인터페이스)에만 의존한다. 이를 통해 **DIP(Dependency Inversion Principle)** 가 자연스럽게 적용된다.

### 핵심 구성 요소

**Port**
- 인터페이스. Application의 경계를 정의한다.
- **Inbound Port**: Application이 외부에 노출하는 인터페이스 (예: `ProductSyncUseCase`)
- **Outbound Port**: Application이 외부에 요구하는 인터페이스 (예: `EbayApiPort`, `ProductRepositoryPort`)

**Adapter**
- Port의 구현체. 외부와 Application을 연결한다.
- **Primary Adapter (Inbound)**: 외부에서 Application을 호출하는 구현체 (예: Worker, Batch, REST Controller)
- **Secondary Adapter (Outbound)**: Outbound Port의 실제 구현체 (예: eBay HTTP Client, JPA Repository)

**Application Layer**
- 순수 비즈니스 로직만 존재
- 외부(인프라, DB, HTTP)를 몰라야 한다

## 동작 원리

### 구조 흐름

```
Primary Adapter (Worker/Batch/Controller)
        ↓
Inbound Port (UseCase 인터페이스)
        ↓
Application Layer (순수 비즈니스 로직)
        ↓
Outbound Port (Repository/ExternalService 인터페이스)
        ↓
Secondary Adapter (JPA / HTTP Client / Redis)
```

### 레이어드 아키텍처와의 차이

| | 레이어드 | 헥사고날 |
|--|---------|---------|
| 의존 방향 | 위 → 아래 (단방향) | 외부 → 내부 (DIP 적용) |
| 테스트 | 외부 의존성 Mock 필요 | Application Layer 독립 테스트 가능 |
| 인프라 교체 | 코드 수정 필요 | Adapter만 교체하면 됨 |

### 도입이 타당한 경우

1. **외부 의존성이 복잡하고 변동 가능성이 높을 때**
   - 예: SFTP → API 전환, Router → Kafka 전환 가능성
2. **비즈니스 로직이 충분히 복잡할 때**
   - 단순 CRUD라면 오버엔지니어링
3. **단위 테스트 커버리지가 중요할 때**
   - 인터페이스 격리 → Mock 없이 도메인 테스트 가능

> [!tip] 오케스트레이션 레이어
> Worker처럼 비즈니스 로직 없이 여러 컴포넌트를 조율만 하는 레이어는 테스트할 비즈니스 로직이 없다. 실제 로직은 각 컴포넌트에서 이미 테스트되므로 오케스트레이션 레이어 테스트는 생략 가능하다.

## 코드 예시

### 실전 적용 구조 (eBay 연동 시스템)

| 역할 | 컴포넌트 |
|------|---------|
| Primary Adapter | 국가별 Worker, Batch |
| Inbound Port | `ProductSyncUseCase` |
| Application Layer | 카테고리 매핑, 가격 변환, 등급 매핑 |
| Outbound Port | `EbayApiPort`, `ProductRepositoryPort` |
| Secondary Adapter | eBay HTTP Client, JPA Repository |

**도입 배경**: Service 레이어에 비즈니스 로직 + eBay HTTP 호출 + 토큰 발급 + 필터링 조건 혼재 → 분리 불가능한 구조

**도입 효과**
- Application Layer 외부 의존성 제거 → 단위 테스트 100% (Application + Service Layer 기준)
- 국가 추가 시 Worker(Primary Adapter) 1개만 추가하면 되는 확장 구조

### 오케스트레이션 레이어 예시

```typescript
// Worker = 오케스트레이션 레이어 (비즈니스 로직 없음)
async consume(goodsNo: string) {
    const product = await this.productRepository.findById(goodsNo); // 위임
    const mapped = await this.mapper.map(product);                  // 위임
    await this.ebayApi.register(mapped);                            // 위임
}
```

> [!warning] 오버엔지니어링 주의
> 단순 CRUD 서비스에 헥사고날 아키텍처를 적용하면 불필요한 추상화 레이어만 증가한다. 비즈니스 로직의 복잡도와 외부 의존성 변동 가능성을 기준으로 도입 여부를 판단해야 한다.
