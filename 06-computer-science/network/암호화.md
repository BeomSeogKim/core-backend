# 암호화

## 핵심 개념

**암호화(Encryption)** — 데이터를 제3자가 읽을 수 없는 형태로 변환하는 것. 네트워크에서는 전송 중 탈취를 방지하기 위해 사용한다.

## 동작 원리

### 대칭키 암호화 (Symmetric Key)

```
송신자                         수신자
  |                               |
  |--- [같은 키로 암호화] -----→  |
  |                               |--- [같은 키로 복호화]
  |
  문제: 키를 어떻게 안전하게 전달하지?
        전달 과정에서 탈취되면 끝
```

- 암호화 · 복호화에 **동일한 키** 사용
- **속도 빠름** — 실제 데이터 전송에 사용
- **키 배포 문제** — 키 전달 과정 자체가 보안 취약점

### 비대칭키 암호화 (Asymmetric Key)

```
  공개키(Public Key)   — 누구에게나 공개. 암호화에 사용.
  개인키(Private Key)  — 나만 보관.       복호화에 사용.

  [기밀성] 공개키로 암호화 → 개인키로만 복호화
  ┌──────────┐   공개키 암호화   ┌──────────┐
  │  송신자  │ ─────────────→  │  수신자  │
  └──────────┘                  └──────────┘
                                  개인키 복호화 (나만 읽을 수 있음)

  [인증] 개인키로 암호화 → 공개키로 복호화
  ┌──────────┐   개인키 암호화   ┌──────────┐
  │  서버    │ ─────────────→  │  누구나  │
  └──────────┘                  └──────────┘
                                  공개키로 복호화 → "이게 서버가 보낸 것 맞음" 증명
                                  = 전자서명 (Digital Signature)
```

| 방향 | 목적 |
|---|---|
| 공개키로 암호화 → 개인키로 복호화 | **기밀성** — 수신자만 읽을 수 있음 |
| 개인키로 암호화 → 공개키로 복호화 | **인증/무결성** — 전자서명, 보낸 이 증명 |

### TLS Handshake (HTTPS)

대칭키(빠름)와 비대칭키(안전)의 단점을 보완한 조합.

```
Client                                    Server
  |                                          |
  |--- ClientHello (지원 암호화 목록) -----→ |
  |←-- ServerHello + 인증서(공개키 포함) --- |
  |                                          |
  | [인증서 검증]                            |
  |  CA 목록으로 서명 확인                   |
  |                                          |
  |--- 대칭키를 공개키로 암호화해서 전송 ---→ |
  |                  개인키로 복호화 → 대칭키 획득
  |                                          |
  |←====== 이후 대칭키로 암호화 통신 ======→ |
```

**CA (Certificate Authority, 인증 기관)**

```
[인증서 발급]
  서버 → CA에 공개키 + 도메인 정보 제출
  CA → 개인키로 서명 → 인증서 발급

[브라우저 검증]
  브라우저에 신뢰할 CA 목록 내장
  인증서의 CA 서명을 CA 공개키로 검증 → 진짜면 통과
```

## 대칭키 vs 비대칭키 비교

| | 대칭키 | 비대칭키 |
|---|---|---|
| 키 개수 | 1개 (공유) | 공개키 + 개인키 |
| 속도 | **빠름** | 느림 |
| 키 배포 | 위험 (탈취 가능) | **안전** |
| 용도 | 실제 데이터 전송 | 키 교환, 전자서명 |
| 알고리즘 예시 | AES, DES | RSA, ECC |

## HTTPS = TLS Handshake + 대칭키 통신

```
[전체 흐름]
  1. 비대칭키로 대칭키 안전하게 교환  (TLS Handshake)
  2. 교환된 대칭키로 실제 데이터 암호화 전송  (빠른 통신)
```

## 코드 예시

```java
// Spring Security — HTTPS 강제
@Configuration
public class SecurityConfig {
    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http.requiresChannel(channel ->
            channel.anyRequest().requiresSecure()  // HTTP → HTTPS 리다이렉트
        );
        return http.build();
    }
}
```

```java
// Java — AES 대칭키 암호화 예시
SecretKey key = KeyGenerator.getInstance("AES").generateKey();
Cipher cipher = Cipher.getInstance("AES");
cipher.init(Cipher.ENCRYPT_MODE, key);
byte[] encrypted = cipher.doFinal("plaintext".getBytes());
```
