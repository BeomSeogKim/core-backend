# 프로세스 스케줄링

## 핵심 개념

**프로세스 스케줄링**은 OS가 어떤 프로세스/스레드를 다음에 실행할지 결정하는 정책이다. CPU라는 한정된 자원을 여러 프로세스에 효율적으로 배분하기 위해 필요하다.

## 동작 원리

### 스케줄러와 디스패처

CPU 스케줄링은 **스케줄러**와 **디스패처**, 두 가지 구성 요소가 협력하여 동작한다.

| 구성 요소 | 역할 |
|-----------|------|
| **스케줄러 (Scheduler)** | 다음에 어떤 프로세스를 실행할지 **결정** (정책) |
| **디스패처 (Dispatcher)** | 스케줄러의 결정을 **실제로 수행** (메커니즘) |

디스패처가 수행하는 작업:
1. **[[컨텍스트-스위칭]]** — 이전 프로세스 상태 저장 + 다음 프로세스 상태 복원
2. **커널 모드 → 유저 모드 전환**
3. 새 프로세스의 적절한 위치로 **점프 (PC 복원)**

### 스케줄링 발생 시점

```
        ┌──────────┐
        │   New    │
        └────┬─────┘
             ↓ ⑤ 프로세스 생성
        ┌──────────┐    ① I/O 요청 등    ┌──────────┐
        │  Ready   │←───────────────────│ Waiting  │
        └────┬─────┘   ③ I/O 완료        └──────────┘
             ↓                                ↑
        ┌──────────┐──────────────────────────┘
        │ Running  │        ① Running → Waiting
        └────┬─────┘
             ↓ ④ 프로세스 종료
        ┌──────────┐
        │Terminated│
        └──────────┘
```

| 시점 | 전이 | 선점/비선점 |
|------|------|------------|
| ① I/O 요청 등 블로킹 | Running → Waiting | 비선점 (자발적) |
| ② Time quantum 만료, 높은 우선순위 도착 | Running → Ready | 선점 (강제) |
| ③ I/O 완료 | Waiting → Ready | 선점 가능 |
| ④ 프로세스 종료 | Running → Terminated | 비선점 (자발적) |
| ⑤ 새 프로세스 생성 | New → Ready | 선점 가능 |

**비선점 스케줄링**에서는 ①, ④ (프로세스가 자발적으로 CPU를 놓는 경우)에만 스케줄링이 발생하고, **선점 스케줄링**에서는 모든 시점에서 스케줄링이 가능하다.

### 스케줄링 평가 기준

| 기준 | 설명 |
|------|------|
| **CPU 활용률** | CPU가 놀지 않고 일하는 비율 |
| **처리량 (Throughput)** | 단위 시간당 완료된 프로세스 수 |
| **응답 시간 (Response Time)** | 요청 후 첫 응답까지 걸리는 시간 |
| **대기 시간 (Waiting Time)** | Ready Queue에서 대기한 총 시간 |
| **반환 시간 (Turnaround Time)** | 제출 후 완료까지 총 시간 |

### 비선점 (Non-preemptive) 스케줄링

실행 중인 프로세스의 CPU를 강제로 뺏지 않는다. 프로세스가 자발적으로 CPU를 반환(I/O 요청, 종료)할 때까지 기다린다.

### 선점 (Preemptive) 스케줄링

OS가 실행 중인 프로세스의 CPU를 강제로 뺏을 수 있다. [[커널|타이머 인터럽트]]가 이를 가능하게 한다.

#### SRTF (Shortest Remaining Time First)

SJF의 선점 버전. 새 프로세스가 도착할 때마다 **남은 실행 시간**을 비교하여, 현재 실행 중인 프로세스보다 짧으면 CPU를 빼앗는다.

```
P1(7ms, 도착:0), P2(4ms, 도착:2), P3(1ms, 도착:4), P4(4ms, 도착:5)

P1 ██ P2 ██ P3 █ P2 ██ P4 ████ P1 █████
0    2    4   5   7     11      16

→ 도착 시점마다 남은 시간 비교 후 전환
```

- SJF보다 평균 대기 시간을 더 줄일 수 있음
- Context Switching이 빈번하게 발생
- 긴 프로세스에 **Starvation** 발생 가능

#### Round Robin

각 프로세스에 동일한 **Time Quantum**을 부여하고, 시간이 다 되면 강제 전환.

```
Time Quantum = 4ms
P1(24ms), P2(3ms), P3(3ms)

P1 ████ P2 ███ P3 ███ P1 ████ P1 ████ P1 ████ P1 ████ P1 ████
0     4     7    10    14    18    22    26    30

→ P2, P3가 빠르게 완료됨. 응답성이 좋음.
```

- 현대 OS의 기본 스케줄링
- Time Quantum이 너무 크면 → FCFS와 동일
- Time Quantum이 너무 작으면 → [[컨텍스트-스위칭]] 오버헤드 증가

#### Priority Scheduling

우선순위가 높은 프로세스를 먼저 실행.

- **Starvation** — 낮은 우선순위 프로세스가 영원히 실행되지 못하는 문제
- **Aging** — 대기 시간이 길어지면 우선순위를 점진적으로 높여서 starvation 해결

#### Multilevel Queue

프로세스를 성격별로 여러 큐에 분류하고, 각 큐마다 다른 스케줄링 알고리즘을 적용한다.

```
[높은 우선순위]  시스템 프로세스 큐      → FCFS
                 대화형(interactive) 큐  → Round Robin
                 배치(batch) 큐          → SJF
[낮은 우선순위]
```

- 큐 간 우선순위가 고정되어 있어 상위 큐가 비어야 하위 큐가 실행됨
- 프로세스가 한번 배정되면 **다른 큐로 이동할 수 없다** → **Starvation** 발생 가능
- 이 한계를 개선한 것이 MLFQ

#### Multilevel Feedback Queue (MLFQ)

현대 OS가 실제로 사용하는 방식. 여러 우선순위 큐를 두고, 프로세스의 행동에 따라 큐 간 이동.

```
Queue 0 (최고 우선순위, 짧은 Time Quantum)
  │ Time Quantum 다 쓰면 강등 ↓
Queue 1 (중간 우선순위, 중간 Time Quantum)
  │ Time Quantum 다 쓰면 강등 ↓
Queue 2 (최저 우선순위, 긴 Time Quantum)
```

- **IO bound 프로세스** — CPU를 조금 쓰고 바로 I/O 대기 → 상위 큐에 머무름 → 응답성 좋음
- **CPU bound 프로세스** — Time Quantum을 다 씀 → 하위 큐로 강등 → 긴 시간 배정
- **Aging** — 하위 큐에서 오래 대기한 프로세스는 상위 큐로 승격시켜 Starvation 방지
- 프로세스 성격을 미리 분류할 필요 없이 동작을 보고 자동으로 적절한 큐에 배치되는 효과

### 스케줄링 알고리즘 비교

| 알고리즘 | 선점 여부 | 장점 | 단점 |
|----------|-----------|------|------|
| FCFS | 비선점 | 단순 | Convoy Effect |
| SJF | 비선점 | 평균 대기 시간 최소 | 실행 시간 예측 어려움 |
| SRTF | 선점 | SJF보다 평균 대기 시간 개선 | Context Switching 빈번, Starvation |
| Round Robin | 선점 | 응답성 좋음 | Time Quantum 튜닝 필요 |
| Priority | 선점/비선점 | 중요 작업 우선 | Starvation |
| Multilevel Queue | 선점 | 성격별 분류, 큐별 최적 알고리즘 | 큐 간 이동 불가, Starvation |
| MLFQ | 선점 | 자동 분류, 균형, Aging | 구현 복잡 |
