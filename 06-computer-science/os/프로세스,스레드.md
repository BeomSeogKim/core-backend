# 프로세스와 스레드

## 핵심 개념

**프로세스(Process)**는 실행 중인 프로그램으로, 독립적인 메모리 공간(Code, Data, Stack, Heap)을 할당받는다. **스레드(Thread)**는 프로세스 내에서 실행되는 동작 단위로, Stack을 제외한 나머지 메모리 영역을 공유한다.

> [!note] 프로세스 vs 스레드 핵심 차이
> 프로세스는 독립적인 메모리 공간을 가지며 서로 격리된다. 스레드는 같은 프로세스 내에서 Code, Heap, Data 영역을 공유하므로 통신이 빠르지만, 동시 접근 시 Race Condition이 발생할 수 있다.

## 동작 원리

### 프로세스 (Process)
실행중인 프로그램 (program in execution) 이라는 뜻으로 실행파일 형태로 존재하던 program이 memory에 적재되어 CPU에 의해 실행되는 것

#### 메모리 적재
메모리는 CPU가 직접 접근할 수 있는 컴퓨터 내부의 기억장치.

프로그램이 CPU에서 실행되기 위해선 해당 내용이 메모리에 적재된 상태여야만 한다.

프로세스에 할당되는 메모리 공간은 Code, Data, Stack, Heap 4개의 영역으로 이루어져 있으며, 프로세스 별로 독립적으로 할당을 받는다.

#### CPU의 연산과 PC Register
**PC Register**에는 다음에 실행될 코드 (명령어, instruction)의 주소값이 저장되어 있다.

즉, 메모리에 적재되어있는 프로세스의 `code`영역의 명령어 중 다음 번 연산에서 읽어야 할 명령어의 주소값을 PC Register가 순차적으로 가리키고,

해당 명령어를 읽어와 CPU가 연산을 하게 되면 process가 실행이 되는 방식.

### 스레드 (Thread)
한 프로세스 내에서 실행되는 동작 (function)의 단위.

각 스레드는 속해있는 프로세스의 Stack 메모리를 제외한 나머지 (code, heap, data) 영역을 공유한다.

#### 메모리 영역별 역할

| 영역 | 내용 | 스레드 공유 여부 |
|------|------|----------------|
| Code | 실행할 명령어 (바이트코드) | 공유 |
| Data | 전역 변수, static 변수 | 공유 |
| Heap | 동적 할당 메모리 (new, malloc) | 공유 |
| Stack | 지역 변수, 함수 호출 정보, return address | 독립 |

> [!warning] Data 영역 공유와 Race Condition
> Data 영역을 여러 스레드가 공유하기 때문에, static 변수에 동시 접근 시 Race Condition이 발생할 수 있다.

#### 스택 메모리 & PC Register
스레드가 함수를 호출하기 위해서는 인자 전달, Return Address 저장, 함수 내 지역변수 저장등을 위한 독립적인 스택 메모리 공간이 필요하다.

결과적으로 스레드는 stack memory 영역은 독립적으로 할당받고, code, heap, data 영역은 공유하는 형태를 지닌다.

스레드도 프로세스와 마찬가지로 각자 다음에 실행될 코드의 주소값들을 알아야 스레드별로 별개로 작업을 진행할 수 있기 때문에 스레드별로 할당을 받는다.

### 멀티 프로세스 (Multi Process)
두개 이상의 프로세스가 동시에 실행되는 것을 말한다. 이 때, 프로세스들은 CPU와 메모리를 공유한다.

memory의 경우 여러 프로세스들이 각자의 메모리 영역을 차지해 동시에 적재된다. 반면 하나의 CPU는 매 순간 하나의 프로세스만 처리할 수 있다.

CPU의 처리속도가 빠르기에, 짧은 시간 (수 ms) 사이에 여러 프로세스들이 CPU에서 번갈아 실행되는 방식으로 진행된다.

### 멀티 스레드 (Multi Thread)
두개 이상의 스레드가 동시에 실행되는 것을 말한다.

### 멀티프로그래밍 (Multiprogramming)
여러 프로그램을 메모리에 올려두고, 한 프로세스가 I/O 대기 상태에 들어가면 CPU를 다른 프로세스에게 할당하는 방식.

- **목적**: CPU 유휴 시간(idle time) 제거 -> CPU 활용률(Utilization) 극대화
- **전환 시점**: 현재 프로세스가 I/O 대기에 들어갈 때만 전환 (수동적)
- **배경**: 1950~60년대 CPU가 고가의 자원이던 시대에 등장한 개념

### 멀티 프로그래밍, 멀티 태스킹, 멀티 프로세싱

관점
- 멀티 프로그래밍 : I/O 대기 시 다른 프로세스로 전환해 CPU 활용률을 높이는가?
- 멀티 태스킹 : 여러 프로세스와 여러 스레드가 아주 짧게 쪼개진 cpu time을 나눠 갖는가?
- 멀티 스레딩 : 프로세스가 두개 이상의 스레드를 사용하는 가?
- 멀티 프로세싱 : 두개 이상의 프로세서나 코어를 활용하는 시스템인가?

> [!tip] 멀티프로그래밍 vs 멀티태스킹 핵심 차이
> 멀티프로그래밍은 I/O 대기 시에만 전환하여 CPU 활용률을 높이고, 멀티태스킹은 Time Quantum으로 OS가 강제 전환하여 응답성을 향상시킨다.

#### 멀티프로그래밍 vs 멀티태스킹

| | 멀티프로그래밍 | 멀티태스킹 |
|---|---|---|
| 전환 시점 | I/O 대기 시에만 | 시간 할당량(Time Quantum) 초과 or I/O 대기 |
| 목적 | CPU 활용률 극대화 | 응답성(Responsiveness) 향상 |
| 특징 | 프로세스가 CPU를 자발적으로 양보 | OS가 강제로 CPU를 빼앗음 (Preemptive) |

|                           | 멀티 태스킹 | 멀티 스레딩 | 멀티 프로세싱 |
|---------------------------|--------|--------|---------|
| 싱글 코어 CPU, 싱글 스레드 프로세스 두개 | O      | X      | X       |
| 싱글 코어 CPU, 듀얼 스레드 프로세스 한개 | O      | O      | X       |
| 듀얼 코어 CPU, 싱글 스레드 프로세스 두개 | X      | X      | O       |
| 듀얼 코어 CPU, 듀얼 스레드 프로세스 한개 | X      | O      | O       |
| 듀얼 코어 CPU, 듀얼 스레드 프로세스 두개 | O      | O      | O       |

#### 핵심 요약

| 개념 | 한 줄 정의 |
|------|-----------|
| 멀티프로그래밍 | I/O 대기 시 다른 프로세스 실행 -> CPU 활용률 극대화 |
| 멀티태스킹 | Time Quantum으로 CPU 강제 전환 -> 응답성 향상 |
| 멀티스레딩 | 하나의 프로세스에 다수의 스레드가 동작 |
| 멀티프로세싱 | 여러 CPU 코어로 진짜 병렬(True Parallelism) 처리 |
