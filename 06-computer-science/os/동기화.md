# 동기화 (Synchronization)

## 핵심 개념

멀티스레드 환경에서 여러 스레드가 공유 자원에 동시 접근하면 **Race Condition**이 발생할 수 있다. 이를 방지하기 위해 **Critical Section**에 대한 접근을 제어하는 것이 **동기화(Synchronization)**다.

- **Race Condition** — 여러 스레드가 공유 데이터에 동시 접근하고, 최소 하나가 **쓰기**를 할 때 타이밍에 따라 결과가 달라지는 상황
- **Critical Section** — 공유 자원에 접근하는 코드 영역. 하나의 스레드만 실행해야 하는 구간
- **Synchronization** — 여러 스레드가 동시에 실행되어도 공유 데이터의 일관성을 보장하는 메커니즘

## 동작 원리

### Race Condition 발생 과정

```java
// 공유 변수
int count = 0;

// Thread-A, Thread-B 동시 실행
count++;  // 한 줄이지만 실제로는 3단계
```

`count++`의 실제 동작:

```
1. READ  — count 값을 CPU 레지스터로 읽기
2. ADD   — 레지스터 값에 1 더하기
3. WRITE — 결과를 count에 저장
```

> [!warning] 읽기만 하면 문제없다
> 모든 스레드가 공유 자원을 **읽기만** 한다면 race condition은 발생하지 않는다. **최소 하나의 스레드가 쓰기**를 해야 문제가 된다.

### Critical Section 문제 해결 조건

Critical section 문제를 올바르게 해결하려면 세 가지 조건을 모두 만족해야 한다:

| 조건 | 설명 | 위반 시 문제 |
|------|------|-------------|
| **Mutual Exclusion (상호 배제)** | 한 번에 하나의 스레드만 critical section 진입 | Race condition |
| **Progress (진행)** | critical section이 비어있으면 대기 중인 스레드가 반드시 진입 가능 | Deadlock |
| **Bounded Waiting (한정된 대기)** | 요청한 스레드는 유한 시간 내 반드시 진입 | Starvation |

### 동기화 메커니즘

#### Mutex (Mutual Exclusion)

**열쇠가 1개**인 잠금 장치. 하나의 스레드만 critical section에 진입.

```java
mutex.lock();
try {
    count++;  // critical section
} finally {
    mutex.unlock();  // 반드시 같은 스레드가 해제
}
```

- lock/unlock의 **주체가 동일한 스레드**여야 한다
- 소유권(ownership) 개념이 있다

#### Semaphore

**카운터 기반** 동기화. 최대 N개 스레드가 동시 진입 가능.

```
Semaphore(N):
  - wait() (= P 연산): 카운터 감소. 0이면 대기
  - signal() (= V 연산): 카운터 증가. 대기 중인 스레드 깨움
```

- `Semaphore(1)` = **Binary Semaphore** — 동작은 mutex와 유사하지만 **다른 스레드가 signal 가능**
- `Semaphore(N)` = **Counting Semaphore** — 리소스 풀(커넥션 풀 등) 관리에 활용

#### Mutex vs Semaphore

| | Mutex | Semaphore |
|---|---|---|
| 동시 진입 | 1개 | N개 |
| 소유권 | 있음 (같은 스레드가 unlock) | 없음 (다른 스레드가 signal 가능) |
| 용도 | 단일 자원 보호 | 제한된 리소스 풀 관리 |

#### Monitor

**Mutex + Condition Variable**을 하나로 묶은 고수준 동기화 구조. 언어/런타임 차원에서 제공한다.

Java의 `synchronized`가 대표적인 monitor 구현:

```java
synchronized (lock) {
    count++;  // mutual exclusion 보장
}
```

- 개발자가 lock/unlock을 직접 관리할 필요 없음
- 블록 진입 시 자동 lock, 벗어나면 자동 unlock
- `wait()`, `notify()`, `notifyAll()`로 condition variable 역할 수행

## Deadlock (교착상태)

두 개 이상의 스레드가 서로 상대방이 점유한 자원을 기다리며 **무한히 대기**하는 상태.

### 발생 조건 4가지 (전부 만족해야 발생)

| 조건 | 설명 |
|------|------|
| **Mutual Exclusion (상호 배제)** | 하나의 자원은 한 스레드만 점유 가능 |
| **Hold and Wait (점유 대기)** | 자원을 가진 채로 다른 자원을 기다림 |
| **No Preemption (비선점)** | 점유한 자원은 점유한 스레드만 해제 가능 (강제로 뺏을 수 없음) |
| **Circular Wait (순환 대기)** | 대기 관계가 원형으로 형성 |

### 해결 방법

| 전략 | 방식 | 특징 |
|------|------|------|
| **예방 (Prevention)** | 4조건 중 하나를 아예 못 만들게 설계 | 순환 대기 깨기가 가장 실용적 |
| **회피 (Avoidance)** | 자원 할당 전에 안전 상태인지 검사 | 은행원 알고리즘 (Banker's Algorithm) |
| **탐지 (Detection)** | 발생을 허용하되 주기적으로 탐지 | 자원 할당 그래프 순환 검사 |
| **회복 (Recovery)** | 탐지 후 프로세스 강제 종료 또는 자원 선점 | 탐지와 함께 사용 |

4가지 조건 중 하나를 깨면 Deadlock을 방지할 수 있다. 실무에서 가장 흔한 방법은 **순환 대기를 깨는 것** — 락 획득 순서를 일관되게 정하면 순환이 발생하지 않는다. 더 자세한 내용은 [[데드락]] 참고.
