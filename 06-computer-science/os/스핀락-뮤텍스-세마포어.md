# 스핀락 · 뮤텍스 · 세마포어

## 핵심 개념

[[동기화]]의 세 가지 메커니즘. 모두 Critical Section에 하나(또는 N개)의 스레드만 진입하도록 제어하지만, **대기 방식**과 **소유권** 유무에서 근본적으로 다르다.

> [!note] 뮤텍스는 바이너리 세마포어가 아니다
> `Semaphore(1)`은 한 번에 하나의 스레드만 진입한다는 점에서 Mutex와 유사해 보이지만, **소유권(Ownership)** 개념이 없어 동치가 아니다.

## 동작 원리

### Spinlock — busy-wait

락을 획득하지 못한 스레드가 while 루프를 돌며 **CPU를 점유한 채 대기**한다.

```
while (lock == LOCKED) {
    // 아무것도 안 하고 계속 체크 (spinning)
}
lock = LOCKED;  // 락 획득
```

```
Thread-A [=====작업중=====]
Thread-B [spin spin spin spin] [=====작업중=====]
CPU      [AAAAAAAAAAAAAAAAAABBBBBBBBBBBBBBBBBBBB]
                              ^Thread-B도 CPU 점유
```

- **장점**: 컨텍스트 스위칭 없음 → 락 점유 시간이 짧을 때 유리
- **단점**: 대기 중에도 CPU 낭비 → 락 점유 시간이 길면 손해

### Mutex — sleep/wake

락을 획득하지 못한 스레드를 **sleep** 시킨다. 락이 해제되면 OS가 깨운다.

```
Thread-A: lock() → 작업 → unlock()
Thread-B: lock() 실패 → sleep → (A가 unlock) → wake up → lock() 획득
```

- sleep/wake up 시 **컨텍스트 스위칭** 발생 (커널 개입)
- **소유권 있음** — lock을 건 스레드만 unlock 가능

```java
mutex.lock();
try {
    // critical section
} finally {
    mutex.unlock();  // 반드시 같은 스레드가 호출
}
```

### Semaphore — 카운터 기반

내부 카운터로 N개의 스레드가 동시 진입하도록 제어한다.

```
Semaphore(N):
  wait()   (P 연산): 카운터 감소. 0이면 sleep
  signal() (V 연산): 카운터 증가. 대기 중 스레드 wake up
```

- **소유권 없음** — wait()한 스레드가 아닌 다른 스레드가 signal() 가능
- `Semaphore(1)` = Binary Semaphore — 하나만 진입하지만, 다른 스레드가 signal 가능
- `Semaphore(N)` = Counting Semaphore — 커넥션 풀 등 제한된 리소스 관리

## 소유권(Ownership) 차이

Mutex의 소유권이 없다면 다른 스레드가 임의로 unlock할 수 있어 동시성 문제가 발생할 수 있다.

| | Mutex | Binary Semaphore |
|---|---|---|
| 카운터 | — | 1 |
| 소유권 | **있음** | **없음** |
| unlock/signal 주체 | lock을 건 스레드만 | 아무 스레드나 가능 |
| 용도 | 단일 자원 보호 | 순서 제어, 이벤트 신호 |

### Semaphore의 소유권 없음 → 순서 제어에 활용

wait()과 signal()의 주체가 달라도 되므로, **스레드 실행 순서 제어(Ordering)**에 자연스럽게 사용된다.

```
Thread-A (작업 1):               Thread-B (작업 2):
  작업 실행...                      semaphore.wait()   ← A 완료 전까지 대기
  semaphore.signal()  ──────────►   작업 실행...
```

Mutex로는 불가능하다. lock을 건 스레드(A)만 unlock할 수 있으므로, B가 signal 역할을 할 수 없다.

## 비교 요약

| | Spinlock | Mutex | Semaphore |
|---|---|---|---|
| 대기 방식 | busy-wait (CPU 점유) | sleep (컨텍스트 스위칭) | sleep |
| 소유권 | 있음 | 있음 | **없음** |
| 동시 진입 수 | 1 | 1 | N |
| 주요 용도 | 짧은 임계구역, 커널 내부 | 단일 자원 보호 | 리소스 풀, 실행 순서 제어 |

> [!tip] 언제 무엇을 쓸까
> - 락 점유 시간이 매우 짧고 컨텍스트 스위칭 비용이 더 클 때 → **Spinlock**
> - 하나의 공유 자원을 특정 스레드가 독점적으로 보호할 때 → **Mutex**
> - 제한된 리소스 풀 관리 or 스레드 간 실행 순서를 맞춰야 할 때 → **Semaphore**
