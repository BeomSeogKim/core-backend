# 프로세스 상태

## 핵심 개념

**프로세스 상태(Process State)**는 프로세스가 생성부터 종료까지 거치는 생명주기를 나타낸다. OS는 이 상태를 기반으로 [[프로세스-스케줄링|스케줄링]]을 수행하고, 자원을 효율적으로 배분한다.

## 동작 원리

### OS 프로세스 5가지 상태

| 상태 | 설명 |
|------|------|
| **New** | 프로세스 생성 중 (PCB 할당, 메모리 적재 준비) |
| **Ready** | 메모리에 올라와 CPU 할당만 기다리는 상태 |
| **Running** | CPU를 점유하고 명령어를 실행 중 |
| **Waiting (Blocked)** | I/O 완료 등 외부 이벤트를 기다리는 상태 |
| **Terminated** | 실행 완료, 자원 정리 중 |

### 상태 전이 다이어그램

```
                    admitted                          exit
           ┌──────────────────┐                 ┌──────────────┐
           │                  ▼                 │              ▼
        ┌─────┐          ┌─────────┐  dispatch  ┌─────────┐  ┌────────────┐
        │ New │          │  Ready  │ ─────────→ │ Running │  │ Terminated │
        └─────┘          └─────────┘            └─────────┘  └────────────┘
                              ▲                   │   │
                              │   I/O completed   │   │ I/O request
                              │  ┌─────────┐      │   │
                              └──│ Waiting │←─────┘   │
                                 └─────────┘          │
                              ▲                       │
                              │   timer interrupt     │
                              │   (preemption)        │
                              └───────────────────────┘
```

### 주요 전이 조건

| 전이 | 조건 | 비고 |
|------|------|------|
| New → Ready | OS가 프로세스 승인 (admitted) | PCB 생성, 메모리 할당 |
| Ready → Running | **디스패처**가 CPU 할당 (dispatch) | [[프로세스-스케줄링]] 결정에 따라 |
| Running → Ready | Time quantum 만료, 높은 우선순위 프로세스 도착 | 선점 스케줄링 |
| Running → Waiting | I/O 요청, `wait()` 호출 등 | 자발적 CPU 반납 |
| Waiting → Ready | I/O 완료, 이벤트 발생 | 다시 CPU 할당 대기 |
| Running → Terminated | 프로세스 실행 완료 또는 강제 종료 | 자원 회수 |

## Java 스레드 상태 (Thread.State)

Java는 OS 프로세스 상태를 기반으로 하되, **6가지**로 더 세분화한다.

### 6가지 상태

| 상태 | 설명 | 진입 조건 |
|------|------|----------|
| **NEW** | 스레드 객체 생성, `start()` 호출 전 | `new Thread()` |
| **RUNNABLE** | 실행 가능 상태 (OS의 Ready + Running) | `start()` 호출 |
| **BLOCKED** | lock 획득 대기 | `synchronized` 블록 진입 시 lock 획득 실패 |
| **WAITING** | 무기한 대기, 명시적 깨움 필요 | `wait()`, `join()`, `park()` |
| **TIMED_WAITING** | 시간 제한 대기 | `sleep(ms)`, `wait(ms)`, `join(ms)` |
| **TERMINATED** | 실행 완료 | `run()` 종료 또는 예외 발생 |

### BLOCKED vs WAITING

둘 다 대기 상태지만 **깨어나는 조건**이 다르다.

| 구분 | BLOCKED | WAITING |
|------|---------|---------|
| **원인** | `synchronized` lock 획득 실패 | `wait()`, `join()`, `park()` 호출 |
| **깨어나는 조건** | lock을 가진 스레드가 lock release → **자동** | 다른 스레드가 `notify()`, `notifyAll()`, `unpark()` → **명시적** |
| **CPU 사용** | 사용 안 함 (대기 큐에서 대기) | 사용 안 함 |
| **장애 분석** | BLOCKED 다수 → **lock 경합** 문제 | WAITING 다수 → **깨워주는 로직** 미동작 문제 |

> [!tip] Thread Dump 분석
> 실무에서 thread dump를 떴을 때 스레드 상태 분포를 보면 병목 원인을 빠르게 파악할 수 있다.
> - BLOCKED 다수 → 특정 lock에 대한 경합 (어떤 스레드가 lock을 오래 잡고 있는지 확인)
> - WAITING 다수 → notify/unpark가 호출되지 않는 로직 버그
> - TIMED_WAITING 다수 → sleep 과다 또는 타임아웃 설정 확인

### OS ↔ Java 상태 매핑

```
┌─────────────────────────────────────────────────────────┐
│                    OS 프로세스 상태                        │
├──────────┬─────────────────┬──────────────┬─────────────┤
│   New    │  Ready+Running  │   Waiting    │ Terminated  │
└────┬─────┴────────┬────────┴──────┬───────┴──────┬──────┘
     │              │               │              │
     ▼              ▼               ▼              ▼
┌────────┐  ┌────────────┐  ┌─────────────┐  ┌──────────┐
│  NEW   │  │  RUNNABLE  │  │  BLOCKED    │  │TERMINATED│
└────────┘  └────────────┘  │  WAITING    │  └──────────┘
                            │  TIMED_WAIT │
                            └─────────────┘
             Java Thread.State
```

- Java의 **RUNNABLE** = OS의 Ready + Running → JVM은 실제 CPU 점유 여부를 구분하지 않음. OS가 결정하는 영역이기 때문
- OS의 **Waiting** = Java의 BLOCKED + WAITING + TIMED_WAITING → Java가 대기 원인별로 3가지로 세분화

## 코드 예시

```java
// Java 스레드 상태 확인
Thread t = new Thread(() -> {
    synchronized (lock) {
        // RUNNABLE → lock 획득 성공
    }
});

System.out.println(t.getState()); // NEW

t.start();
System.out.println(t.getState()); // RUNNABLE

// 다른 스레드가 lock을 잡고 있다면
// t.getState() → BLOCKED

// wait() 호출 후
// t.getState() → WAITING

// sleep(1000) 호출 후
// t.getState() → TIMED_WAITING
```
