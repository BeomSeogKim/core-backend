## Platform Thread (OS Thread)

JVM의 기본 스레드. OS 스레드와 **1:1 매핑**.

```
JVM Thread  ←→  OS Thread  ←→  CPU Core
   (1)              (1)
```

### 특성
- OS 커널이 스케줄링 담당
- 스택 메모리: ~1MB (OS 영역, 고정 할당)
- 생성 비용: 높음 (OS syscall 필요)
- 최대 개수: 수천 개 (메모리 한계)

```java
// Platform Thread 생성
Thread thread = new Thread(() -> { ... });
thread.start();  // OS에 스레드 생성 요청 (syscall)
```

## Context Switching 비용

CPU 코어는 한 번에 하나의 스레드만 실행. 여러 스레드를 번갈아 실행하기 위해 **컨텍스트 스위칭** 발생.

### 전환 과정

```
[Thread-A 실행 중]
  CPU 레지스터: [A의 PC, SP, 범용 레지스터들]

OS 스케줄러가 Thread-B로 전환 결정:
  1. User Mode → Kernel Mode (syscall)
  2. Thread-A의 CPU 레지스터 → TCB-A에 저장
  3. 스케줄러가 Thread-B 선택
  4. TCB-B에서 Thread-B의 레지스터 복원
  5. Kernel Mode → User Mode
  6. Thread-B 실행 재개
```

### 비용 요소

**1. Kernel Mode Transition**
- User Space ↔ Kernel Space 전환 자체에 비용
- ~1~10 μs per switch

**2. CPU 캐시 Cold Miss**
```
Thread-A 실행 중:
  L1 Cache: [A의 데이터 ★★★]  ← 따뜻한 캐시

Thread-B로 스위칭:
  L1 Cache: [B의 데이터로 교체]  ← A 데이터 사라짐

Thread-A 재개:
  L1 Cache miss → L2 → L3 → RAM
  L1: ~1ns vs RAM: ~100ns  (100배 차이)
```

**3. TLB Miss** (프로세스 간 전환 시)
- Translation Lookaside Buffer 무효화
- 가상주소 → 물리주소 변환 캐시 초기화

### 스레드 수와 성능의 관계

```
요청 10,000개 / Platform Thread 10,000개:
  - 메모리: 10,000 × 1MB = 10GB (스택만)
  - CPU 8코어: 1초에 수천 번 컨텍스트 스위칭
  - 실제 작업 시간 < 스위칭 오버헤드
  → "컨텍스트 스위칭 지옥"
```

## Preemptive vs Cooperative 스케줄링

### Preemptive (Platform Thread)
OS가 강제로 스레드를 중단하고 다른 스레드로 전환.

```
Thread-A: ████████|강제중단|████████
Thread-B:         |████████|
                  ↑ OS 타임슬라이스 만료
```

- 개발자가 제어 불가
- 공정한 CPU 분배
- 언제든 중단될 수 있음 → 공유 자원 보호 필요 (synchronized)

### Cooperative (Kotlin Coroutine)
코루틴이 **자발적으로** 실행권을 양보 (`suspend` 지점에서).

```
Coroutine-A: ████████|suspend|재개████████
Coroutine-B:         |████████|
                     ↑ awaitSingle() 등 suspend point
```

- 개발자가 suspend 지점 명시
- 스레드 전환 없이 같은 스레드에서 스위칭 가능
- suspend point 없는 CPU 작업은 다른 코루틴 실행 막음 → `yield()` 필요

## Virtual Thread (Project Loom, JDK 21)

### 개념

JVM이 관리하는 경량 스레드. Platform Thread(carrier thread) 위에서 N:M 실행.

```
Virtual Thread 1 ─┐
Virtual Thread 2 ──┤ Platform Thread 1 (carrier)
Virtual Thread 3 ─┘

Virtual Thread 4 ─┐
Virtual Thread 5 ──┤ Platform Thread 2 (carrier)
Virtual Thread 6 ─┘
```

### 동작 방식

```
Virtual Thread가 blocking I/O 호출:
  → JVM이 감지
  → Virtual Thread를 carrier thread에서 언마운트
  → Virtual Thread 상태를 힙에 저장
  → carrier thread는 다른 Virtual Thread 실행

I/O 완료:
  → JVM이 Virtual Thread를 carrier thread에 마운트
  → 실행 재개
```

개발자는 그냥 blocking 코드를 작성하면 JVM이 투명하게 처리.

```java
// 그냥 blocking 코드 작성
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    executor.submit(() -> {
        var response = restTemplate.getForObject(url, String.class);  // blocking OK
        return process(response);
    });
}
```

### 특성

| | Platform Thread | Virtual Thread |
|---|---|---|
| 관리 주체 | OS | JVM |
| 스케줄링 | OS Preemptive | JVM (carrier thread pool) |
| 메모리 | ~1MB 고정 스택 | ~1KB~ 가변 (힙) |
| 최대 개수 | 수천 | 수백만 |
| blocking I/O | 스레드 점유 | JVM이 언마운트 |
| 프로그래밍 모델 | blocking | blocking (투명하게 처리) |

## Coroutine vs Virtual Thread

```
                  Kotlin Coroutine          Virtual Thread
관리 주체          Kotlin 런타임              JVM
전환 방식          Cooperative (suspend)     JVM이 투명하게 처리
개발자 코드         suspend 명시 필요          기존 blocking 코드 그대로
I/O 라이브러리      비동기 API 필요 (WebClient)  동기 API 사용 가능 (RestTemplate)
상태 저장           Continuation 객체 (힙)     가상 스택 (힙)
Backpressure       지원 (Reactor와 통합)      미지원
```

### 각각 언제 선택하나?

**Virtual Thread 선택:**
- 기존 blocking 코드베이스 마이그레이션
- JDBC, 동기 HTTP 클라이언트 사용 중
- 간단한 I/O 위주 서비스
- Spring MVC + `spring.threads.virtual.enabled=true`

**Coroutine 선택:**
- Backpressure 필요 (스트리밍, 이벤트 파이프라인)
- 이미 WebFlux/Reactor 기반 코드베이스
- Kotlin 코드베이스 (언어 수준 통합)
- 복잡한 async 파이프라인 구성

## Virtual Thread 주의사항: Pinning

`synchronized` 블록 안에서 blocking I/O 시 carrier thread에 고정(pin)됨.

```java
// Pinning 발생 - 피해야 함
synchronized(lock) {
    jdbcTemplate.query(...);  // blocking 중 synchronized 보유
    // → carrier thread가 언마운트 못하고 고정됨
    // → Virtual Thread의 이점 사라짐
}

// 해결책: ReentrantLock 사용
ReentrantLock lock = new ReentrantLock();
lock.lock();
try {
    jdbcTemplate.query(...);  // OK - JVM이 언마운트 가능
} finally {
    lock.unlock();
}
```

JDK 24에서 synchronized Pinning 문제 해결 예정.
