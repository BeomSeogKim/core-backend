# 가상 메모리 (Virtual Memory)

## 핵심 개념

**가상 메모리(Virtual Memory)**는 각 프로세스에게 "자기만의 독립된 메모리 공간"이 있다고 착각하게 만드는 OS의 메모리 관리 기법이다. 실제 물리 RAM은 하나지만, OS가 가상 주소 → 물리 주소로 매핑하여 프로세스 간 메모리를 격리한다.

## 동작 원리

### 가상 주소 공간

각 프로세스는 독립된 가상 주소 공간을 가진다. 같은 가상 주소라도 프로세스마다 다른 물리 주소에 매핑된다:

```
프로세스 A (가상 주소)          물리 RAM            프로세스 B (가상 주소)
┌──────────────┐          ┌──────────────┐      ┌──────────────┐
│ 0x0000 ──────│──────────│→ 0x5000      │      │ 0x0000 ──────│──┐
│ 0x1000 ──────│──────┐   │              │  ┌───│── 0x1000     │  │
│ 0x2000 ──────│──┐   │   │              │  │   │              │  │
└──────────────┘  │   └──→│  0x8000      │  │   └──────────────┘  │
                  │       │              │  │                      │
                  └──────→│  0x2000      │  │                      │
                          │              │←─┘                      │
                          │  0xA000      │←────────────────────────┘
                          └──────────────┘

A의 0x0000 → 물리 0x5000
B의 0x0000 → 물리 0xA000  (같은 가상 주소, 다른 물리 주소)
```

### 페이지와 페이지 테이블

가상 메모리는 **페이지(Page)** 단위로 관리한다 (보통 4KB).

- **가상 페이지** — 가상 주소 공간을 4KB 단위로 나눈 것
- **물리 프레임** — 물리 RAM을 4KB 단위로 나눈 것
- **페이지 테이블** — 프로세스마다 존재. 가상 페이지 → 물리 프레임 매핑 정보를 저장

```
프로세스 A의 페이지 테이블:

가상 페이지  →  물리 프레임  |  유효 비트
─────────────────────────────────────
  Page 0    →  Frame 5     |    1 (RAM에 있음)
  Page 1    →  Frame 8     |    1
  Page 2    →  Disk 주소    |    0 (RAM에 없음, 디스크에 있음)
  Page 3    →  Frame 2     |    1
```

유효 비트가 0이면 해당 페이지가 RAM에 없다는 뜻 → **Page Fault** 발생 → 디스크에서 RAM으로 로드.

### 주소 변환: MMU와 TLB

매번 메모리에 접근할 때마다 가상 주소 → 물리 주소 변환이 필요하다. 이것을 하드웨어가 처리한다:

- **MMU (Memory Management Unit)** — CPU 칩 내의 하드웨어. 페이지 테이블을 참조해서 주소 변환 수행
- **TLB (Translation Lookaside Buffer)** — **MMU 내부**의 캐시. 최근 Page→Frame 변환 결과를 캐싱하여 매번 메모리의 페이지 테이블을 조회하지 않아도 되게 함

TLB hit rate는 보통 95% 이상이라 대부분의 메모리 접근은 빠르게 처리된다.

### 컨텍스트 스위칭과의 관계

프로세스가 전환되면 페이지 테이블이 바뀐다. 이때:

1. **MMU의 페이지 테이블 포인터**를 새 프로세스의 것으로 변경
2. **TLB flush** — 이전 프로세스의 매핑 캐시를 무효화
3. 직후 **TLB miss 연속 발생** → 성능 저하

이것이 프로세스 [[컨텍스트-스위칭]]이 스레드 컨텍스트 스위칭보다 비싼 핵심 이유 중 하나다. 같은 프로세스 내 스레드 간 전환은 주소 공간이 동일하므로 TLB flush가 불필요하다.

### Demand Paging

프로세스의 모든 페이지를 물리 메모리에 올리지 않는다. **실제로 접근하는 페이지만 물리 메모리에 로드**하고, 나머지는 디스크에 둔다. 이를 통해 물리 메모리보다 큰 가상 주소 공간을 사용할 수 있다.

### Page Fault 처리 과정

물리 메모리에 없는 페이지에 접근하면 **Page Fault**가 발생한다:

```
1. CPU가 가상 주소로 접근
2. MMU가 페이지 테이블 확인 → 유효 비트 = 0 → Page Fault 발생
3. OS가 디스크에서 해당 페이지를 물리 메모리로 로드
4. 페이지 테이블 업데이트 (유효 비트 = 1, 프레임 번호 기록)
5. 중단됐던 명령어 재실행
```

### 페이지 교체 알고리즘

물리 메모리가 가득 찬 상태에서 Page Fault가 발생하면, 기존 페이지를 내보내고 새 페이지를 로드해야 한다:

| 알고리즘 | 방식 | 특징 |
|----------|------|------|
| **FIFO** | 먼저 들어온 것부터 교체 | 단순하지만 비효율 |
| **LRU** | 가장 오래 안 쓴 것부터 교체 | 가장 널리 사용, 시간적 지역성 활용 |
| **LFU** | 사용 빈도가 낮은 것부터 교체 | 빈도 기반 |
| **Optimal** | 앞으로 가장 안 쓸 것을 교체 | 이론적 최적, 구현 불가 (미래 예측 불가) |
