# Connection Pool

## 핵심 개념

**Connection Pool**은 DB 커넥션(TCP 연결)을 미리 생성하여 풀에 보관하고, 요청 시 빌려주고 반납받는 방식으로 재사용하는 기법이다. 매 요청마다 커넥션을 새로 생성하는 비용을 제거한다.

## 동작 원리

### 커넥션 생성 비용

DB 커넥션을 맺을 때 발생하는 작업:
1. **TCP 3-way Handshake** — 네트워크 연결
2. **DB 인증** — 사용자 자격 검증
3. **세션 할당** — DB 내부 메모리 할당

이 과정을 매 요청마다 반복하면 응답 시간이 길어지고 DB 부하가 증가한다.

### Connection Pool 동작

```
애플리케이션                   Connection Pool                  DB
                          ┌──────────────────────┐
[요청 1] ── 빌림 ─────→   │ conn1 ■ (사용중)      │ ────→ DB
[요청 2] ── 빌림 ─────→   │ conn2 ■ (사용중)      │ ────→ DB
[요청 3] ── 대기...       │ conn3 □ (대기중)      │
                          │ conn4 □ (대기중)      │
[요청 1] ── 반납 ─────→   │ conn1 □ (대기중)      │
[요청 3] ── 빌림 ─────→   │ conn1 ■ (사용중)      │ ────→ DB
                          └──────────────────────┘
                           ■ 사용중  □ 대기중
```

1. 애플리케이션 시작 시 설정된 수만큼 **커넥션을 미리 생성**
2. 요청이 오면 풀에서 **유휴 커넥션을 빌려줌**
3. 작업 완료 후 커넥션을 **풀에 반납** (연결 끊지 않음)
4. 풀이 비어있으면 새 커넥션 생성 또는 **대기** (timeout까지)

### HikariCP — Spring Boot 기본 Connection Pool

```yaml
# application.yml
spring:
  datasource:
    hikari:
      maximum-pool-size: 10      # 최대 커넥션 수
      minimum-idle: 5            # 유휴 상태로 유지할 최소 커넥션 수
      connection-timeout: 30000  # 커넥션 대기 최대 시간 (ms)
      idle-timeout: 600000       # 유휴 커넥션 유지 시간 (ms)
      max-lifetime: 1800000      # 커넥션 최대 수명 (ms)
```

### 풀 사이즈 결정

풀 크기가 너무 작으면 커넥션 대기가 발생하고, 너무 크면 DB에 과부하가 걸린다.

```
최적 크기 ≈ (CPU 코어 수 * 2) + 유효 디스크 수
```

> [!note] 커넥션 풀과 스레드 풀의 관계
> 웹 서버의 스레드 풀 크기가 커넥션 풀보다 크면, 남는 스레드는 커넥션을 기다리며 대기한다. 일반적으로 커넥션 풀 크기 >= 동시 DB 접근 스레드 수로 설정한다.
