# 데이터베이스 락

## 핵심 개념

**DB 락**은 동시에 여러 트랜잭션이 같은 데이터에 접근할 때 데이터 일관성을 보장하기 위한 동시성 제어 메커니즘이다. 공유 락/배타 락의 DB 레벨 락과, 낙관적/비관적 락의 애플리케이션 레벨 전략으로 나뉜다.

## 동작 원리

### 공유 락 (Shared Lock, S Lock)

**읽기(SELECT)** 작업에 사용. 여러 트랜잭션이 동시에 S Lock을 획득할 수 있다.

```sql
SELECT * FROM orders WHERE id = 1 FOR SHARE;  -- S Lock 획득
```

### 배타 락 (Exclusive Lock, X Lock)

**쓰기(INSERT, UPDATE, DELETE)** 작업에 사용. 다른 어떤 락과도 호환되지 않는다.

```sql
SELECT * FROM orders WHERE id = 1 FOR UPDATE;  -- X Lock 획득
```

### 락 호환성

| | S Lock 보유 중 | X Lock 보유 중 |
|---|---|---|
| S Lock 요청 | **호환** (동시 읽기 가능) | **대기** |
| X Lock 요청 | **대기** | **대기** |

읽기끼리는 동시에 가능하지만, 쓰기가 끼면 대기한다.

### 락 범위

| 범위 | 설명 | 동시성 |
|------|------|--------|
| **Row Lock** | 행 단위 잠금 | 높음 |
| **Page Lock** | 페이지(여러 행) 단위 | 중간 |
| **Table Lock** | 테이블 전체 | 낮음 |

InnoDB는 기본적으로 **Row Lock**을 사용하여 동시성을 높인다.

### 낙관적 락 vs 비관적 락

DB 자체의 락이 아닌, 애플리케이션 레벨에서의 동시성 제어 전략.

| | 낙관적 락 (Optimistic) | 비관적 락 (Pessimistic) |
|---|---|---|
| 가정 | 충돌이 **거의 없을 것** | 충돌이 **자주 발생할 것** |
| 방식 | **버전 컬럼**으로 충돌 감지 | **DB 락**으로 선점 |
| 충돌 시 | 예외 → 애플리케이션에서 재시도 | 대기 (락 해제까지) |
| 장점 | 락 없어서 동시성 높음 | 데이터 정합성 확실 |
| 단점 | 충돌 잦으면 재시도 비용 | 락 대기로 성능 저하 |
| 적합 | 읽기 많은 환경 | 쓰기 경쟁 심한 환경 |

#### 낙관적 락 — 버전 기반

```java
@Entity
public class Order {
    @Version
    private Long version;  // JPA가 자동 관리
}
```

```sql
-- UPDATE 시 버전 체크
UPDATE orders SET status = 'DONE', version = 2
WHERE id = 1 AND version = 1;
-- 영향받은 행이 0이면 → 다른 트랜잭션이 먼저 수정함 → OptimisticLockException
```

#### 비관적 락 — DB 락 선점

```java
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT o FROM Order o WHERE o.id = :id")
Order findByIdForUpdate(@Param("id") Long id);
// → SELECT ... FOR UPDATE 실행 → X Lock 획득
```
