## 구조화된 동시성 (Structured Concurrency)

코루틴은 **부모-자식 계층 구조**를 가진다.
- 부모 코루틴이 취소되면 모든 자식 코루틴도 취소
- 자식 코루틴이 모두 완료되어야 부모가 완료
- 자식의 실패가 부모로 전파 (coroutineScope 기준)

## coroutineScope + async: 병렬 처리

### async의 동작

```kotlin
coroutineScope {
    val deferred1 = async { fetchDetail(id) }   // 즉시 코루틴 시작, Deferred 반환
    val deferred2 = async { fetchPrice(id) }    // 즉시 코루틴 시작, Deferred 반환
    // 여기까지는 두 요청이 이미 동시에 진행 중

    val detail = deferred1.await()  // 완료 대기 (suspend)
    val price = deferred2.await()   // 이미 완료됐으면 즉시 반환
}
```

- `async {}`: 코루틴을 즉시 시작하고 `Deferred<T>` 반환
- `await()`: 결과가 올 때까지 suspend (blocking 아님)
- `awaitAll()`: 여러 Deferred를 한 번에 대기

```kotlin
// 여러 요청 병렬 처리
val results = coroutineScope {
    chunks.map { chunk ->
        async { fetchChunk(chunk) }
    }.awaitAll()
}
// 총 지연 = max(각 청크 응답시간)  ← N × latency 아님
```

### awaitAll() vs map { await() }

```kotlin
// 권장: awaitAll() - 하나가 실패하면 나머지 즉시 취소
val results = deferreds.awaitAll()

// 비권장: map { await() } - 순서대로 하나씩 대기 (사실상 직렬)
val results = deferreds.map { it.await() }
```

## coroutineScope vs supervisorScope

### coroutineScope: 자식 실패 → 전체 취소

```kotlin
try {
    coroutineScope {
        val a = async { successTask() }
        val b = async { failTask() }    // 예외 발생!
        // b가 실패하면 a도 즉시 취소됨
        // coroutineScope 자체도 예외 전파
        awaitAll(a, b)
    }
} catch (e: Exception) {
    // 전체 실패 처리
}
```

**적합한 상황**: 모든 작업이 성공해야 의미 있는 경우 (트랜잭션처럼)

### supervisorScope: 자식 실패 → 해당 자식만 실패

```kotlin
supervisorScope {
    val deferreds = items.map { item ->
        async { processItem(item) }
    }

    deferreds.map { deferred ->
        try {
            deferred.await()
        } catch (e: Exception) {
            log.error("항목 처리 실패: $e")
            null  // 실패한 항목은 null로 처리
        }
    }.filterNotNull()
}
```

**적합한 상황**: 부분 실패를 허용하고 나머지는 계속 처리해야 하는 경우

| | coroutineScope | supervisorScope |
|---|---|---|
| 자식 실패 시 | 모든 자식 취소 + 예외 전파 | 해당 자식만 실패 |
| 실패 추적 | 불가 (전체 롤백) | 가능 (개별 try-catch) |
| 적합한 경우 | All-or-nothing | 부분 성공 허용 |

## onErrorReturn vs supervisorScope: 실전 선택

### onErrorReturn (Reactor 레벨)

```kotlin
coroutineScope {
    chunks.map { chunk ->
        async {
            productClient.getProducts(chunk)
                .onErrorReturn(emptyList())  // Reactor 파이프라인에서 에러 흡수
                .awaitSingle()               // 코루틴은 항상 emptyList() 수신
        }
    }.awaitAll()
}
```

에러 흐름:
```
외부 API 500 에러
  → onErrorReturn(emptyList())  ← Reactor에서 흡수
  → awaitSingle()은 emptyList() 수신 (예외 없음)
  → async {} 정상 완료
  → coroutineScope 예외 없음
```

### 선택 기준

| | onErrorReturn | supervisorScope + try-catch |
|---|---|---|
| 에러 처리 레이어 | Reactor 스트림 | 코루틴 레벨 |
| 실패 항목 식별 | 불가 | 가능 |
| 에러 로깅/재시도 | 어려움 | 용이 |
| 코드 복잡도 | 낮음 | 높음 |
| 적합한 상황 | fallback 값으로 충분 | 실패 추적/재시도 필요 |

## 취소 (Cancellation)

코루틴 취소는 협력적(cooperative)이다. suspend point에서만 취소 감지.

```kotlin
val job = launch {
    repeat(1000) { i ->
        if (!isActive) return@launch  // 취소 확인 (CPU-bound 작업 시)
        delay(100)                    // suspend point → 여기서 취소 감지
        println(i)
    }
}

job.cancel()  // 취소 신호 전송
job.join()    // 취소 완료 대기
```

### CancellationException

취소는 `CancellationException`으로 전파됨. 일반 예외와 다르게 처리해야 함.

```kotlin
try {
    delay(1000)
} catch (e: CancellationException) {
    throw e  // 반드시 재던져야 함! 삼키면 취소가 안 됨
} catch (e: Exception) {
    // 일반 에러 처리
}
```

## 실전 패턴: Scatter-Gather

분산 처리 후 결과 수집하는 패턴

```kotlin
suspend fun fetchAllProducts(productIds: List<Long>): List<Product> {
    return coroutineScope {
        productIds
            .chunked(25)                           // 25개씩 청크
            .map { chunk ->
                async {
                    productClient.getProducts(chunk)
                        .onErrorReturn(emptyList()) // 부분 실패 허용
                        .awaitSingle()
                }
            }
            .awaitAll()                            // 모든 청크 완료 대기
            .flatten()                             // List<List<Product>> → List<Product>
    }
}
```
