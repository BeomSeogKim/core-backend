# 모니터 (Monitor)

## 핵심 개념

**Monitor**란 Race Condition을 방지하기 위해 Java가 제공하는 동기화 메커니즘으로, Lock과 Wait Set을 결합하여 Critical Section에 대한 상호 배제와 조건 대기를 함께 처리한다.

## 왜 필요한가

여러 스레드가 공유 자원에 동시에 접근하면 **Race Condition**이 발생한다.

```
count++ 는 실제로 3단계:
  1. 메모리에서 값 읽기
  2. +1 연산
  3. 메모리에 쓰기

두 스레드가 동시에 1단계를 실행하면 → 덮어쓰기 발생
결과: 2번 증가해야 할 값이 1번만 증가
```

이를 막으려면 **Critical Section**을 만들어 한 번에 하나의 스레드만 접근하도록 제한해야 한다.

## 모니터 구성 요소

```
┌─────────────────────────────────┐
│           Monitor               │
│                                 │
│  ┌─────────┐    ┌───────────┐   │
│  │  Lock   │    │  Wait Set │   │
│  └─────────┘    │ (Cond Var)│   │
│                 └───────────┘   │
│  ┌──────────────────────────┐   │
│  │      Entry Queue         │   │
│  └──────────────────────────┘   │
└─────────────────────────────────┘
```

| 구성 요소 | 역할 |
|---|---|
| **Lock** | Critical Section 진입 권한. 한 번에 하나의 스레드만 보유 가능 |
| **Entry Queue** | 락 획득을 대기 중인 스레드들의 큐 |
| **Wait Set** | `wait()` 호출 후 조건을 대기 중인 스레드들의 집합 |
| **Condition Variable** | Wait Set과 짝을 이뤄 "어떤 조건을 기다리는지" 표현 |

## 동작 원리

### 전체 흐름

```
스레드 진입 시도
    │
    ▼
락 획득 가능? ──No──→ Entry Queue 대기
    │
   Yes
    │
    ▼
조건 확인 (while)
    │
    ├──NG──→ wait() 호출
    │            │
    │            ├─ 락 반납
    │            └─ Wait Set 진입
    │
   OK
    │
    ▼
작업 수행
    │
    ▼
notify() / notifyAll()
    │
    └─→ Wait Set 스레드 → Entry Queue 이동 → 락 재획득 → 조건 재확인
    │
    ▼
락 반납 (synchronized 블록 종료)
```

### `wait()` 동작

- 호출 조건: **락을 보유한 스레드만** 호출 가능 (아니면 `IllegalMonitorStateException`)
- 효과: 락 반납 + Wait Set 진입 (두 가지 동시에 발생)

### `notify()` / `notifyAll()` 동작

- `notify()`: Wait Set에서 스레드 하나를 Entry Queue로 이동 (어떤 스레드인지 보장 없음)
- `notifyAll()`: Wait Set의 모든 스레드를 Entry Queue로 이동
- 깨어난 스레드는 **즉시 실행되지 않음** → Entry Queue에서 락 재획득 후 실행

## 코드 예시

### synchronized 메서드

```java
public class Counter {
    private int count = 0;

    // 락 = this (Counter 인스턴스)
    public synchronized void increment() {
        count++;
    }
}
```

### wait() / notify() 패턴

```java
public class BoundedBuffer {
    private final Queue<Integer> buffer = new LinkedList<>();
    private final int capacity;

    public BoundedBuffer(int capacity) {
        this.capacity = capacity;
    }

    public synchronized void produce(int item) throws InterruptedException {
        while (buffer.size() == capacity) {  // if 아닌 while!
            wait();  // 락 반납 + Wait Set 진입
        }
        buffer.add(item);
        notifyAll();
    }

    public synchronized int consume() throws InterruptedException {
        while (buffer.isEmpty()) {  // if 아닌 while!
            wait();
        }
        int item = buffer.poll();
        notifyAll();
        return item;
    }
}
```

### `if` 대신 `while`을 써야 하는 이유

```
notify()는 어떤 스레드를 깨울지 보장하지 않는다.
조건에 맞지 않는 스레드가 깨어날 수 있으므로
깨어난 후 반드시 조건을 재확인해야 한다.

if → 조건 재확인 없이 통과 → 버그
while → 조건 불만족 시 다시 wait() → 안전
```
