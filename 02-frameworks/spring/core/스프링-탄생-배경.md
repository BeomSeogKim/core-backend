# 스프링 프레임워크 탄생 배경

## 핵심 개념

Spring Framework는 **EJB(Enterprise JavaBeans)** 의 복잡성과 침투적 설계에 대한 반발로 탄생했다. Rod Johnson이 2002년 저서 "Expert One-on-One J2EE Design and Development"에서 EJB 없이도 좋은 엔터프라이즈 애플리케이션을 만들 수 있음을 증명하면서 시작됐다.

## EJB의 문제점

### 1. 프레임워크 종속 (침투적 프로그래밍)

EJB는 비즈니스 클래스가 EJB 스펙 인터페이스를 **반드시 구현**해야 했다. 비즈니스 로직보다 프레임워크 코드가 더 많아지는 문제가 발생했다.

```java
// EJB 시대 — 비즈니스 로직이 EJB 코드에 오염됨
public class OrderBean implements SessionBean {  // 강제 구현
    private SessionContext context;

    public void ejbCreate() { }          // 강제 구현
    public void ejbRemove() { }          // 강제 구현
    public void ejbActivate() { }        // 강제 구현
    public void ejbPassivate() { }       // 강제 구현
    public void setSessionContext(SessionContext ctx) { this.context = ctx; }

    // 실제 비즈니스 로직
    public void createOrder(Order order) {
        // 여기까지 오는 데 얼마나 많은 코드가 필요했는가
    }
}
```

### 2. 테스트 불가

EJB 컨테이너(WebLogic, JBoss 등) 없이는 Bean을 생성할 수도, 테스트할 수도 없었다. 단위 테스트가 사실상 불가능했고, 서버를 올려야만 동작 확인이 가능했다.

### 3. 구현체 교체 어려움

EJB 인터페이스에 강하게 결합되어 있어서 특정 WAS나 기술에서 벗어나기 어려웠다. 벤더 종속(Vendor Lock-in)이 심했다.

## EJB 문제 요약

```
EJB 시대의 3가지 고통
┌──────────────────────────────────────┐
│ 1. 프레임워크 종속                    │
│    → 비즈니스 코드 = EJB 스펙 코드    │
│                                      │
│ 2. 테스트 불가                        │
│    → 컨테이너 없이 테스트 불가         │
│                                      │
│ 3. 구현체 교체 어려움                 │
│    → 벤더 Lock-in                    │
└──────────────────────────────────────┘
```

## Spring의 해결책

### POJO(Plain Old Java Object)

Spring은 비즈니스 클래스가 어떤 프레임워크 인터페이스도 구현하지 않는 **순수 Java 객체**로 작성될 수 있게 했다.

```java
// Spring 시대 — 순수 Java 객체
@Service
public class OrderService {
    private final OrderRepository orderRepository;

    public OrderService(OrderRepository orderRepository) {
        this.orderRepository = orderRepository;
    }

    public void createOrder(Order order) {
        orderRepository.save(order);
    }
}
// EJB 스펙 구현 없음, 컨테이너 없이도 new OrderService()로 테스트 가능
```

### Spring이 제시한 3가지 핵심

| 문제 | Spring 해결책 |
|---|---|
| 프레임워크 종속 | **POJO 기반 개발** — 순수 Java 객체로 비즈니스 로직 작성 |
| 구현체 교체 어려움 | **IoC / DI** — 의존성을 외부에서 주입, 인터페이스 기반 설계 |
| 횡단 관심사 오염 | **AOP** — 트랜잭션, 보안, 로깅을 비즈니스 로직과 분리 |

### "경량 컨테이너"

Spring은 스스로를 **경량 컨테이너(Lightweight Container)**라고 칭했다. EJB의 무거운 컨테이너 없이도 Bean의 생명주기와 의존성을 관리할 수 있음을 강조한 것이다.
